Prevent Duplicate Cron Jobs Across Multiple EC2 Instances

02 Jul 2020 AlexS
bash linux aws DevOps

When you run identical cron jobs on multiple EC2 instances, they can all fire simultaneously — leading to duplicate database entries, redundant deployments, or conflicting file operations. The simplest way to guarantee only one instance executes the job is a shared lock file on an EFS mount that all instances can access.

This guide shows how to implement a lightweight distributed mutex using a bash script and an EFS-backed lock file.

Why Prevent Duplicate Execution?

  • Data integrity — avoid duplicate inserts, updates, or conflicting writes
  • Resource efficiency — don't waste CPU, I/O, or network bandwidth on redundant work
  • Deployment safety — prevent multiple instances from deploying the same artifact concurrently
  • Cost savings — reduce unnecessary compute cycles across your fleet

The Solution

Wrap your cron job logic in a lock-file check. All EC2 instances share the same EFS mount point, so only the first instance to create the lock file proceeds:

#!/bin/sh -xv

# Avoid running multiple cron jobs on different instances...
LOCK_FILE="/mount/efs/cronjob.lock"

if [ ! -f "$LOCK_FILE" ]; then
  # Create lock file
  echo "cronjob" > "$LOCK_FILE"

  # Command(s)...
  echo "Test"

  # Delete lock file when done
  rm "$LOCK_FILE"
fi

Schedule this script via cron on every EC2 instance. Only the one that successfully creates the lock file will execute the payload.

Understanding the Script

LinePurpose
#!/bin/sh -xvRuns with debug tracing (-x) and verbose output (-v) for logging
LOCK_FILE="/mount/efs/cronjob.lock"Defines a shared lock file path on the EFS volume
[ ! -f "$LOCK_FILE" ]Checks if the lock file does NOT exist (i.e., no other instance is running)
echo "cronjob" > "$LOCK_FILE"Creates the lock file to signal other instances to skip
rm "$LOCK_FILE"Removes the lock file after successful execution so the next run can proceed

How It Works

  1. Cron trigger — all EC2 instances fire the script at the same scheduled time
  2. Lock check — each instance checks for the existence of /mount/efs/cronjob.lock
  3. First wins — the first instance to create the lock file proceeds with the payload; the rest skip
  4. Cleanup — the winning instance deletes the lock file when finished, readying it for the next cycle

Make It Robust

The basic approach works, but production environments benefit from extra safety:

Add a Timeout on Lock Removal

Ensure the lock is cleaned up even if the script crashes:

#!/bin/sh -xv

LOCK_FILE="/mount/efs/cronjob.lock"

cleanup() {
  rm -f "$LOCK_FILE"
}

trap cleanup EXIT

if [ ! -f "$LOCK_FILE" ]; then
  echo "cronjob" > "$LOCK_FILE"
  # Your commands here...
  echo "Test"
fi

Add an Age-Based Lock Expiry

Prevent stale locks from blocking future runs:

#!/bin/sh -xv

LOCK_FILE="/mount/efs/cronjob.lock"
MAX_AGE=3600  # 1 hour in seconds

cleanup() { rm -f "$LOCK_FILE"; }
trap cleanup EXIT

if [ ! -f "$LOCK_FILE" ]; then
  echo $$ > "$LOCK_FILE"
  echo "Running job..."
elif [ $(find "$LOCK_FILE" -mmin +60) -gt 0 ]; then
  echo "Stale lock detected, overwriting..."
  echo $$ > "$LOCK_FILE"
  echo "Running job..."
else
  echo "Another instance is running. Skipping."
fi

Customization Tips

  • Per-job locks — use a unique lock filename for each cron job (e.g., /mount/efs/deploy.lock, /mount/efs/backup.lock)
  • Instance identification — write the hostname or instance ID into the lock file for easier debugging
  • AWS alternatives — for more advanced coordination, consider AWS DynamoDB locks or S3 object versioning
  • EFS permissions — ensure all EC2 instances have read/write access to the EFS mount point

A simple shared lock that turns a fleet of cron jobs into a coordinated single-runner. One file, zero duplicates.
