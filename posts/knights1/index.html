Knights Game — A JavaScript Puzzle Where Two Knights Share One Board

12 Mar 2021 AlexS
game javascript

The Knights game is a turn-based puzzle played on an 8×8 chessboard. Two players take turns placing knights using standard L-shaped knight movement. The catch: each cell can only be visited once. When you can no longer make a valid move, the game ends and you lose.

No frameworks, no dependencies beyond jQuery for DOM manipulation — just vanilla JavaScript running entirely client-side with an 8-state move validation array and a history tracker that prevents revisiting cells.

This post explains the rules, how the game engine works, and why building it in pure JS makes it fast and portable.

Rules

  • Play by turns — each round one player moves, then the other
  • Use chess knight movement — move 2 squares in one direction and 1 square perpendicular (the classic L-shape: 8 possible destinations)
  • Visited cells can't be visited again — once a knight lands on a cell, that cell is burned forever
  • Loser is who can't make a movement — when the board fills to the point where no valid knight jump exists from the last position, the current player loses

How the Game Engine Works

Board Initialization

The game starts with an empty 8×8 grid. Every cell is marked as both a valid chess square (black or white alternating) and a clickable target for the first move. This means your very first knight can land anywhere on the board.

Knight Move Validation

Eight possible L-shaped offsets are defined in a single array:

var knightMoves = [
  [-2, 1],
  [-1, 2],
  [1, 2],
  [2, 1],
  [2, -1],
  [1, -2],
  [-1, -2],
  [-2, -1],
];

After each move, the engine loops through these offsets, checks board boundaries (0–7 for both axes), and verifies the target cell hasn't been visited before. Valid cells get highlighted as clickable buttons for the next player.

History Tracking

Every move — coordinates and which figure moved — is pushed into a history array. Before validating any potential destination, the engine scans this list to confirm the cell is fresh. This prevents both knights from ever revisiting ground they've already covered.

Computer Opponent

The computer picks a random valid cell from the buttons array after each player move. Simple, but it forces you to think several moves ahead because the computer will eventually corner you by accident.

Win/Loss Detection

After each pair of moves (player + computer), the engine checks if the buttons array is empty:

  • Player wins — when the player's move exhausts all remaining valid cells (the last possible knight placement)
  • Player loses — when the computer's response blocks every available path forward (no valid knight jump exists from the computer's new position)

Two Game Modes

White Knight Mode

You play as the white knight (♘). You move first, which means you choose where on the board to start. The computer responds as the black knight (♞).

Black Knight Mode

You play as the black knight (♞). The computer moves first as white, picking a random starting cell. This mode is harder because you have no control over the opening position and must adapt immediately.

Why Build It in Pure JavaScript?

  • Zero server dependency — the game runs entirely in the browser with no PHP, no database, no API calls
  • Instant startup — no loading screens, no authentication, just click and play
  • Portable — drops into any HTML page that includes jQuery (the project loads 1.12.4 from a CDN)
  • Deterministic — the same board state always produces the same valid moves, making it easy to debug and extend

Source Code Architecture

The entire game lives in three files:

FilePurpose
index.htmlLoads jQuery + knights.js, provides New Game buttons and a .board div for rendering
knights.cssDefines 40×40 pixel squares, alternating black/white backgrounds, figure sizing (40px font), button cursor styling
knights.jsContains the game state object, move validation engine, computer AI, render loop, and click handlers

The render function rebuilds the entire board DOM on every turn — draw squares, overlay visited figures, highlight valid next moves with data attributes for click binding. This brute-force re-render is fast enough for an 8×8 grid and keeps the code simple.
