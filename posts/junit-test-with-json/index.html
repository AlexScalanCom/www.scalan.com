JUnit Parameterized Tests with JSON Fixtures

27 Sep 2024 AlexS
java springboot

Problem: You need to validate a Spring Boot service method against dozens of input/output scenarios — manually constructing and asserting each case in code quickly becomes unmaintainable. A cleaner approach is to externalize test data into JSON fixture files and drive the tests parameterized.

Solution Overview

Use JUnit 5 Parameterized Tests (@ParameterizedTest + @CsvSource) to load pairs of input/expected JSON files from the classpath. For each pair:

  1. Read the input JSON fixture from src/test/resources.
  2. Deserialize it into your domain object via Jackson ObjectMapper.
  3. Invoke the service method under test.
  4. Serialize the result back to JSON.
  5. Compare against the expected output fixture using JSONAssert.

This pattern keeps test data version-controlled, human-readable, and completely separate from test logic.

Prerequisites

Ensure your project includes these test dependencies (Maven example):

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>

<dependency>
    <groupId>org.skyscreamer</groupId>
    <artifactId>jsonassert</artifactId>
    <scope>test</scope>
</dependency>

Test Class Implementation

package myapp;

import com.fasterxml.jackson.databind.JsonNode;
import com.fasterxml.jackson.databind.ObjectMapper;
import org.json.JSONObject;
import org.junit.jupiter.api.Assertions;
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.CsvSource;
import org.skyscreamer.jsonassert.JSONAssert;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.context.ActiveProfiles;

import java.nio.charset.StandardCharsets;
import java.nio.file.Files;
import java.nio.file.Paths;
import java.util.Objects;

@SpringBootTest(classes = App.class)
@ActiveProfiles({"dev", "db"})
class EventTransformerTest {

    @Autowired
    private MyService transformer;

    private final ObjectMapper objectMapper = new ObjectMapper();

    @ParameterizedTest
    @CsvSource({
        "data/in1.json, data/out1.json",
        "data/in2.json, data/out2.json",
        "data/in3.json, data/out3.json",
        "data/in4.json, data/out4.json"
    })
    void transformServiceIT(String payloadFilename, String responseFilename) throws Exception {

        // 1. Read input fixture from classpath
        String payloadString = new String(
            Files.readAllBytes(Paths.get(
                Objects.requireNonNull(
                    Thread.currentThread()
                        .getContextClassLoader()
                        .getResource(payloadFilename))
                        .toURI())),
            StandardCharsets.UTF_8);

        // 2. Extract "Event" node and deserialize to domain object
        JsonNode eventNode = objectMapper.readTree(payloadString);
        String eventString = eventNode.get("Event").toString();
        InEvent inEvent = objectMapper.readValue(eventString, InEvent.class);

        // 3. Read expected output fixture from classpath
        String correctResponseString = new String(
            Files.readAllBytes(Paths.get(
                Objects.requireNonNull(
                    Thread.currentThread()
                        .getContextClassLoader()
                        .getResource(responseFilename))
                        .toURI())),
            StandardCharsets.UTF_8);

        // 4. Invoke service under test
        OutEvent result = transformer.transform(inEvent);

        // 5. Assert result is valid
        Assertions.assertNotNull(result);
        Assertions.assertNotNull(result.getTxt());

        // 6. Serialize result and compare against expected JSON
        String resultString = objectMapper.writeValueAsString(result);
        JSONAssert.assertEquals(correctResponseString, new JSONObject(resultString), false);
    }
}

How It Works — Step by Step

StepDescription
@SpringBootTestBootstraps the full Spring application context for integration testing.
@ActiveProfilesActivates dev and db profiles to load test-appropriate configuration.
@ParameterizedTest + @CsvSourceDrives the test method once per CSV row — each row specifies an input/expected JSON file pair.
Classpath resource loadingThread.currentThread().getContextClassLoader().getResource(...) resolves files under src/test/resources/.
Jackson deserializationObjectMapper reads the raw JSON, extracts the "Event" field, and maps it to InEvent.
Service invocationtransformer.transform(inEvent) exercises the real business logic.
JSONAssert comparisonJSONAssert.assertEquals(..., false) performs a lenient structural comparison (order-independent) between actual and expected output.

Fixture File Structure

Place all JSON fixtures under src/test/resources/:

src/test/resources/
├── data/
│   ├── in1.json        ← Input: raw JSON with "Event" wrapper
│   ├── out1.json       ← Expected: transformed OutEvent as JSON
│   ├── in2.json
│   ├── out2.json
│   ├── in3.json
│   ├── out3.json
│   └── ...

Example in1.json:

{
  "Event": {
    "id": "evt-001",
    "type": "CLICK",
    "payload": { "x": 120, "y": 340 }
  }
}

Example out1.json:

{
  "txt": "Processed CLICK event evt-001 at (120, 340)",
  "timestamp": 1727000000000
}

Why This Pattern Rocks

  • Scalability — Add a new test case by dropping in two JSON files and one CSV row. Zero Java code changes.
  • Readability — QA team members and non-Java stakeholders can review and author fixtures directly.
  • Maintainability — Fixture files are version-controlled alongside the code, making regression tracking trivial.
  • Flexibility — Switch to @MethodSource or @ValueSource if you need more complex factory-based fixture generation.

Pro Tips

  • Use JSONAssert.assertEquals(..., true) for strict (order-sensitive) comparison when field order matters.
  • Wrap Files.readAllBytes(...) in a utility method to reduce boilerplate across test classes.
  • Consider the JsonUnit library for even richer JSON diff reporting on assertion failures.

This data-driven testing pattern dramatically reduces boilerplate while increasing confidence that your service handles every edge case correctly.
