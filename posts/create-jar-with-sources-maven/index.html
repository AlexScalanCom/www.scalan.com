Package Java Sources into JAR with Maven

07 May 2023 AlexS
java maven

Distributing source JARs alongside compiled artifacts is a best practice for Java projects. Source jars enable IDE debugging, allow downstream consumers to read your code, and are required by public repositories like Maven Central.

Maven provides two approaches: generating a separate -sources.jar with the maven-source-plugin, or bundling sources directly into your main artifact using the maven-assembly-plugin. This guide covers both.

Why Include Sources?

  • Debugging — developers can step into your library's code instead of viewing obfuscated stack traces
  • Readability — IDEs show actual source instead of decompiled stubs
  • Repository compliance — Maven Central and corporate artifact repos often require attached sources
  • Onboarding — new contributors immediately understand your project structure

Approach 1: Separate Source JAR (Recommended)

The standard approach attaches a -sources.jar to your build output. Add the maven-source-plugin to your <build><plugins> section:

<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-source-plugin</artifactId>
            <version>3.3.0</version>
            <executions>
                <execution>
                    <id>attach-sources</id>
                    <goals>
                        <goal>jar-no-fork</goal>
                    </goals>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>

After running mvn package, your target/ directory will contain both artifact.jar and artifact-sources.jar. The source jar is automatically installed to your local repository and deployed alongside the main artifact.

Approach 2: Bundle Sources Inside Main JAR

If you need the raw .java files included as resources inside the compiled JAR (useful for frameworks that load sources at runtime), configure the <resources> block and maven-assembly-plugin:

<build>
    <resources>
        <!-- Standard resources directory -->
        <resource>
            <directory>${basedir}/src/main/resources</directory>
        </resource>
        <!-- Include Java sources as resources -->
        <resource>
            <directory>${basedir}/src/main/java</directory>
        </resource>
    </resources>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-assembly-plugin</artifactId>
            <version>3.7.1</version>
            <executions>
                <execution>
                    <id>assemble-sources</id>
                    <phase>package</phase>
                    <goals>
                        <goal>single</goal>
                    </goals>
                    <configuration>
                        <descriptorRefs>
                            <descriptorRef>jar-with-dependencies</descriptorRef>
                        </descriptorRefs>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>

This approach copies all .java files into the resources/ namespace of your JAR, making them accessible via ClassLoader.getResource().

How It Works (Approach 1)

  1. Build phase — maven-source-plugin runs during the default lifecycle
  2. JAR creation — packages all .java files from src/main/java into a separate archive
  3. Attachment — the source JAR is attached as a classified artifact (sources)
  4. Deployment — both artifact.jar and artifact-sources.jar are installed to ~/.m2/repository

Customization Tips

  • Also generate test sources — add another execution with <goal>test-jar-no-fork</goal>
  • Skip source attachment — run with -Dmaven.source.skip=true when not needed
  • Deploy to remote repo — both jars are automatically pushed with mvn deploy
  • Custom source directories — add <sourcesDirectory> inside the plugin configuration for non-standard layouts

Two solid approaches depending on your distribution needs. Attach sources once, save every developer who depends on your library.
